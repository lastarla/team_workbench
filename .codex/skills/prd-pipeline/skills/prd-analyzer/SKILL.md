# PRD Analyzer

解析 PRD（Product Requirements Document），提取结构化需求规约，消除歧义。

## 输入

- PRD 文档（Markdown、PDF、纯文本、或口头描述）
- 项目上下文（可选：现有代码结构、技术栈、AGENTS.md）

## 输出

- `SPEC.md`：结构化、无歧义、可实现的需求规约

## 详细过程

### Step 1: 结构解析

从 PRD 中提取以下 7 个维度（缺失的维度必须标记为 `[MISSING - 需要澄清]`）：

```markdown
## 1. 核心目标（WHY）
为什么做这个功能？解决什么问题？服务谁？

## 2. 功能需求（WHAT）
每个功能一条，使用 "As a [user], I want [action], so that [benefit]" 格式
或直接描述：系统应该做什么

## 3. 非功能需求（HOW WELL）
- 性能：响应时间、吞吐量、并发数
- 安全：认证、授权、数据保护
- 可维护性：代码风格、测试覆盖率要求
- 兼容性：浏览器/OS/设备支持

## 4. 约束条件（BOUNDARIES）
- 技术栈限制（必须用 X 框架/语言）
- 时间限制
- 不能修改的现有接口/API
- 第三方服务限制

## 5. 验收标准（DONE WHEN）
每个功能需求的具体完成标准
必须是 binary 的（是/否，不是"差不多"）

## 6. 非目标（NOT DOING）
明确声明不做什么（防止 scope creep）

## 7. 依赖与前置条件（NEEDS FIRST）
- 需要什么已经存在（API、数据库、服务）
- 需要什么信息才能开始
```

### Step 2: 歧义检测（苏格拉底式拷问）

对 Step 1 提取的**每个条目**，进行三重检验：

| 检验 | 问题 | 如果"否" |
|------|------|----------|
| **唯一性** | 这个描述是否有且仅有一种实现方式？ | 列出可能的解读 → 请用户选择 |
| **一致性** | 两个独立开发者看到这个描述，会做出相同的东西吗？ | 细化描述直到答案为"是" |
| **可验证性** | "成功"的判断标准是 binary 的吗？ | 将模糊标准转换为具体指标 |

**歧义分类**：

1. **模糊（Vague）**：描述太泛，多种解读均合理
   - 例："界面要好看" → 什么标准？参考什么设计？
   - 行动：请用户提供具体参考或量化标准

2. **矛盾（Contradictory）**：两个需求互相冲突
   - 例："必须实时" + "每分钟同步一次"
   - 行动：指出矛盾，请用户决定优先级

3. **遗漏（Missing）**：关键信息缺失
   - 例：要做登录，但没说用什么认证方式
   - 行动：列出需要的决策点，请用户补充

4. **假设（Assumed）**：隐含假设未明确声明
   - 例：假设用户已经有账号（但 PRD 没说）
   - 行动：将假设显式化，请用户确认

### Step 3: 澄清对话

如果 Step 2 发现歧义，向用户提问。**批量提问**（不要一次一个）：

```markdown
我在分析 PRD 时发现以下需要澄清的点：

**模糊点：**
1. [描述] — 我的理解是 [A]，但也可能是 [B]。请确认哪个正确？
2. ...

**缺失信息：**
3. [需要什么决策] — 选项有 [X/Y/Z]，推荐 [X] 因为 [原因]
4. ...

**需要确认的假设：**
5. 我假设 [假设内容]，这对吗？
6. ...
```

### Step 4: 输出 SPEC.md

使用 `templates/SPEC.template.md` 格式。确保：

- [ ] 每个功能需求有对应的验收标准（1:1 映射）
- [ ] 所有歧义已解决（无 `[MISSING]` 标记）
- [ ] 非目标明确列出（至少 2-3 条）
- [ ] 技术约束与项目实际环境一致
- [ ] 每个需求满足 SMART 原则（Specific, Measurable, Achievable, Relevant, Testable）

## 质量检查清单

完成 SPEC.md 后，自验：

```
□ 没有"应该"、"可能"、"大概"等模糊词
□ 每个需求可以写成自动化测试
□ 非目标足够明确（防止后续 scope creep）
□ 技术约束来自项目实际情况（不是我假设的）
□ 用户已确认所有澄清问题的答案
```

## 示例

**输入**（模糊的 PRD）：
> 做一个用户管理系统，支持注册登录，要安全。

**Step 2 输出**（歧义检测）：
```
模糊点：
1. "用户管理" — 只是注册/登录？还是包含角色管理、权限控制、用户列表管理？
2. "要安全" — 具体安全标准？JWT? OAuth? 2FA? 密码复杂度要求？

缺失信息：
3. 技术栈未指定 — 用什么语言/框架？
4. 数据存储未指定 — SQL? NoSQL? 哪个数据库？
5. 前端形式未指定 — REST API only? 还是带 UI？

假设确认：
6. 我假设这是一个 Web 应用（非移动端），对吗？
7. 我假设不需要社交登录（Google/GitHub OAuth），对吗？
```
