# Auto-Debugger

自动修复 milestone-validator 发现的验证失败。

## 核心原则

> **最小修复，不重构。**
> 目标是让失败的断言通过，不是改善代码质量。
> 修复范围必须限制在失败相关的代码。

## 输入

- `VERDICT.json`：验证结果（失败的断言列表 + 分析）
- 相关代码文件
- 验证命令

## 输出

- 修复后的代码
- 更新的 EVIDENCE.md
- 修复报告

## 修复过程

### Step 1: 失败分类

对每个 FAIL 的断言，分类根因：

| 类别 | 特征 | 修复策略 |
|------|------|---------|
| **逻辑错误** | 代码执行了但结果不对 | 修复算法/条件逻辑 |
| **缺失处理** | 某种情况未处理 | 添加条件分支/error handler |
| **集成问题** | 模块间接口不匹配 | 调整接口调用方式 |
| **配置错误** | 环境/配置导致失败 | 修复配置 |
| **类型错误** | 类型声明与实际不符 | 修正类型定义 |
| **竞态条件** | 时序相关的间歇性失败 | 添加同步机制 |

### Step 2: 修复生成

```
对每个失败断言：
1. 定位相关代码（根据 VERDICT.json 的 analysis）
2. 理解期望行为 vs 实际行为
3. 确定最小修改集
4. 实施修复
5. 本地验证修复有效
```

**修复范围限制**：
- 只修改与失败断言直接相关的代码
- 不重命名变量
- 不重构函数结构
- 不添加新功能
- 不"顺便"修复其他问题

### Step 3: 验证循环

修复后立即运行验证：

```bash
# 1. 验证修复的断言现在通过
[执行该断言的验证命令] → 应该 PASS

# 2. 回归检查（确保修复没引入新问题）
npm run typecheck  → PASS
npm run lint       → PASS
npm test           → ALL PASS（包括之前通过的测试）
```

### Step 4: 循环控制

```
修复轮次 = 0
MAX_ROUNDS = 3

while 有失败断言 AND 修复轮次 < MAX_ROUNDS:
    修复当前失败的断言
    运行验证
    if 全部通过:
        break → SUCCESS
    elif 新的失败出现:
        分析是否是修复引起的副作用
        if 是副作用:
            回退修复 → 尝试不同策略
    修复轮次 += 1

if 修复轮次 >= MAX_ROUNDS:
    标记 BLOCKED → 请求人工介入
```

```markdown

- 失败断言: [N] 个
- 成功修复: [M] 个
- 仍然失败: [N-M] 个
- 修复轮次: [R] / 3

- **验证**: `npm test -- --grep "duplicate email"` → PASS
- **回归**: 全量测试 147/147 PASS
```

## 修复策略库

### 常见模式

**缺少 error handling**：
```
问题: 未处理的异常导致 500
修复: 添加 try-catch + 适当的 HTTP 状态码
```

**类型不匹配**：
```
问题: 函数返回类型与调用方期望不符
修复: 调整返回值结构，或更新调用方解析逻辑
```

**缺少验证**：
```
问题: 输入未验证，无效数据通过
修复: 添加输入验证 + 适当的错误响应
```

**状态管理**：
```
问题: 状态更新遗漏或顺序错误
修复: 确保状态变更的原子性和完整性
```

## 禁止行为

- ❌ 修改测试使其通过（修复代码！）
- ❌ 添加 `@ts-ignore` / `eslint-disable` 压制错误
- ❌ 引入大规模重构
- ❌ 修复范围外的"顺便"改动
- ❌ 在无法理解根因时随机尝试修复
- ❌ 超过 3 轮修复循环继续尝试（必须 BLOCKED）

## 人工介入触发

当以下情况出现时，停止自动修复，请求人工：

1. **修复循环超过 3 轮**：说明问题比预期复杂
2. **修复引入循环依赖**：Fix A → Break B → Fix B → Break A
3. **需要架构决策**：修复需要改变设计方向
4. **外部依赖问题**：第三方服务/API 不符合预期
5. **需求理解错误**：SPEC.md 本身可能有问题
